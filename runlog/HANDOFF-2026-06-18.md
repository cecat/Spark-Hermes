### Departure from plan
Plan wanted **Argo `claudeopus47` primary + vLLM fallback**. Reality: Argo wiring is *correct* at the provider/route level (`compatible-anthropic-endpoint` provider exists with `ANTHROPIC_BASE_URL=http://127.0.0.1:44497`, credential set), but OpenShell 0.0.44's network policy whitelists only `/v1/chat/completions`-style paths for `inference.local` — while Hermes' LiteLLM router emits bare `/chat/completions` (no `/v1/`) and Ollama-style `/api/show` probes. These were denied by OPA even after `policy-add` widened the rules. Confirmed via `openshell policy get --full` that the widened policy v7 was loaded with matching hash — but the in-sandbox enforcer still denied the literal paths. Likely an alpha bug.

**Fix-it-later:** rebuild the sandbox image with a custom Dockerfile (`nemohermes onboard --from <dockerfile>`) embedding a permissive `agents/hermes/policy-additions.yaml`. Filed in RUNLOG as the prime alpha-stack defect.

### Known limitations
1. **Hermes' own Slack adapter is not subscribed in this NemoHermes 0.1.0 build.** The generated `/sandbox/.hermes/config.yaml` only ever enables `platforms.api_server` (per the source of `hermes-config.ts`). `channels add slack` registers a *gateway-level bridge* for cron-deliver purposes but doesn't add `platforms.slack` to the Hermes config. Net effect: inbound (DM-to-Hermes, mention-to-Hermes) won't work; outbound (cron → Slack via the bridge) WILL work.
2. **`/invite @Gandalf`** is required in `#agent-gandalf` before scheduled posts can land — bot can't self-invite without `channels:join`. Until invited, posts will silently fail at the channel-not-member step.
3. Missing scope `groups:read` (regenerate manifest to add if you need private-channel enumeration; public + DM unaffected).

### How we ended up with a token (the long story)
Hermes' `setup.py --auth-url` flow hung consistently with `prompt=consent` — Safari multi-account state was attributing OAuth grants to the wrong `authuser=N` index (silent backend reject, no UI for it). Pivoted to `gog auth add --manual` (Charlie's existing OpenClaw OAuth tool, documented in his OpenClaw-Tutorial), which worked first try. The resulting `gog` keyring entry was exported (`gog auth tokens export ... --out ...`) and reshaped into Hermes' `authorized_user` format. From here on, Hermes manages the token autonomously (refresh hits Google directly, no gog involvement).

## ⚠️ Containment verification (Phase H) — IMPORTANT

The plan's gate H assumed OpenShell's default sandbox policy denies all egress except approved hosts. **It doesn't, in this NemoClaw+Hermes 0.0.44/0.1.0 build.** Tested from inside the sandbox container:

- ✅ Credential isolation: confirmed. Slack tokens, Google client_secret, etc. don't appear in sandbox env — only `xoxb-OPENSHELL-RESOLVE-ENV-...` placeholders. OpenShell's credential proxy injects real values at the network boundary.
- ❌ `curl https://httpbin.org/get` → HTTP 200 (non-approved internet host reachable).
- ❌ `curl http://TAILNET_SPARK_IP:18789/` → HTTP 200 (Tailscale-bound OpenClaw gateway reachable — lateral-movement risk).
- ❌ `curl http://10.0.5.124:22/` and `curl http://host.docker.internal:22/` → TCP connected, got SSH banners (LAN + host SSH reachable at network level; SSH auth itself would still hold).
- ✅ Snapshot `phase-h-baseline` created (v3).

### Recommendation for next session
Add host-side iptables DROP rules on the new `openshell-docker` bridge (`br-a89074d4fc78`, subnet `172.19.0.0/16`) mirroring what `~/code/spark-ai/` already does for the OpenClaw 172.18 bridge:

Until that's done, treat Gandalf as a privileged-network agent, not a contained one, for risk-assessment purposes. The plan's "OpenShell is structural containment" assumption is overstated for the current NemoClaw+Hermes default policy set.
